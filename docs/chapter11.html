1. モジュールとは

Java 9で導入されたモジュール・システム（Project Jigsaw）は、パッケージをまとめたより大きな単位です。モジュールは何に依存するか（requires）と何を公開するか（exports）を明示的に宣言します。

table モジュールシステム導入のメリット

メリット説明
強力なカプセル化exportsしていないパッケージは外部から一切アクセスできない（publicクラスでも不可）
明示的な依存宣言依存関係が module-info.java に可視化されるため、循環依存を検出しやすい
軽量なランタイムjlinkで必要なモジュールだけを含むカスタムJREを作成できる

2. module-info.java

モジュールのルートディレクトリに置く特別なファイルです。このファイル1つでモジュール全体の依存・公開を制御します。

// ファイル名: module-info.java (モジュールのソースルートに配置)
module com.example.myapp {

    // 依存するモジュールを宣言
    requires java.sql;           // JDBCを使うため
    requires java.logging;       // ログAPIを使うため
    requires transitive java.xml; // このモジュールに依存する他モジュールにもjava.xmlが見える

    // 外部に公開するパッケージを宣言
    exports com.example.api;           // 全モジュールに公開
    exports com.example.utils to com.example.client; // 特定モジュールにのみ公開

    // ServiceLoaderの設定
    uses com.example.spi.MyPlugin;   // このインタフェースの実装を検索
    provides com.example.spi.MyPlugin  // このモジュールが実装を提供
        with com.example.impl.MyPluginImpl;

    // リフレクションアクセスを許可
    opens com.example.model;          // 実行時リフレクションを全モジュールに許可
    opens com.example.dto to com.google.gson; // 特定モジュールにのみ許可
}

3. モジュール宣言のキーワード一覧

キーワード役割補足
requires XモジュールXへの依存を宣言コンパイル時・実行時の依存
requires transitive X推移的依存。自分に依存する側にもXが見えるAPIの再公開に使う
requires static Xコンパイル時のみの依存実行時はなくてもよい（アノテーション等）
exports パッケージパッケージを全モジュールに公開publicクラスのみアクセス可
exports パッケージ to MパッケージをモジュールMのみに公開限定公開
opens パッケージパッケージをリフレクションに対して開放実行時のみ。フレームワーク（Spring等）向け
uses インタフェースServiceLoaderでこのインタフェースの実装を検索SPI（サービス提供インタフェース）の消費側
provides I with CインタフェースIの実装Cを提供SPI（サービス提供インタフェース）の提供側

4. コンパイル・実行コマンド

# ディレクトリ構成例
# src/
#   com.example.myapp/
#     module-info.java
#     com/example/myapp/Main.java

# コンパイル: -d でクラス出力先, --module-source-path でソース検索先を指定
javac -d mods --module-source-path src \
      src/com.example.myapp/module-info.java \
      src/com.example.myapp/com/example/myapp/Main.java

# 実行: --module-path でモジュール格納ディレクトリ, -m でモジュール名/メインクラス
java --module-path mods -m com.example.myapp/com.example.myapp.Main

# モジュールJARの作成
jar --create --file=myapp.jar --main-class=com.example.myapp.Main \
    -C mods/com.example.myapp .

# カスタムランタイムイメージの作成 (jlink)
jlink --module-path mods --add-modules com.example.myapp \
      --output myapp-runtime

5. jdeps と ServiceLoader

description jdeps コマンド

jdeps はクラスファイル・JARの依存関係を解析するツールです。モジュールシステムへの移行を支援したり、どのJDKモジュールに依存しているかを確認するために使います。

# JARの依存関係をモジュールレベルで表示
jdeps --module-path mods myapp.jar

# クラスの依存先JDKモジュールを一覧表示
jdeps --list-deps myapp.jar

# モジュール依存グラフをDOT形式で出力
jdeps --module-path mods -dotoutput ./graph myapp.jar

description ServiceLoader

ServiceLoaderは、インタフェースの実装を動的にロードするSPI（サービス提供インタフェース）の仕組みです。モジュールシステムでは uses / provides と組み合わせて使います。

// 1. インタフェースを定義（com.example.spiモジュール）
public interface Greeter {
    String greet(String name);
}

// 2. 実装を提供（com.example.implモジュール のmodule-info.java）
// provides com.example.spi.Greeter with com.example.impl.JapaneseGreeter;

// 3. 消費側（com.example.appモジュール）でServiceLoaderを使って実装を取得
ServiceLoader<Greeter> loader = ServiceLoader.load(Greeter.class);
for (Greeter greeter : loader) {
    System.out.println(greeter.greet("Alice"));
}
// または
loader.findFirst().ifPresent(g -> System.out.println(g.greet("Bob")));
terminal Output こんにちは, Alice！
こんにちは, Bob！
