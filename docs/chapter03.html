1. スレッドの基礎

table スレッド生成方法の比較

方法構文特徴
Threadクラスを継承class MyThread extends Threadシンプルだが他クラスを継承できない
Runnableを実装class MyTask implements Runnable推奨。他クラスも継承できる
ラムダ式new Thread(() -> { ... })最も簡潔。Java 8以降推奨

description スレッドの状態遷移

Javaのスレッドは Thread.State 列挙型で定義された以下の状態を遷移します。

状態説明
NEWnewで生成されたが、まだstart()を呼んでいない
RUNNABLE実行中 または 実行待ちでCPUのスケジューリングを待っている
BLOCKEDsynchronizedブロックのロック取得待ち
WAITINGwait()やjoin()など、無期限に待機中
TIMED_WAITINGsleep(n)など、指定時間だけ待機中
TERMINATED処理が完了し、終了した状態
// ① Threadクラスを継承
class MyThread extends Thread {
    @Override public void run() { System.out.println("extends Thread: " + getName()); }
}
new MyThread().start();

// ② Runnableを実装（推奨）
class MyTask implements Runnable {
    @Override public void run() { System.out.println("implements Runnable: " + Thread.currentThread().getName()); }
}
new Thread(new MyTask()).start();

// ③ ラムダ式（最も簡潔。Java 8以降推奨）
Thread t1 = new Thread(() -> System.out.println("lambda Thread: " + Thread.currentThread().getName()));
t1.start();

// join(): 対象スレッドの完了を待ってから次の処理へ
t1.join();
System.out.println("Main thread continues after t1.");

// Thread.State: スレッドの状態を確認
Thread t2 = new Thread(() -> { try { Thread.sleep(50); } catch (Exception e) {} });
System.out.println("before start: " + t2.getState());  // NEW
t2.start();
System.out.println("after start:  " + t2.getState());  // RUNNABLE
t2.join();
System.out.println("after join:   " + t2.getState());  // TERMINATED
terminal Output Thread running: Thread-0
Main thread continues after t1 finishes.

2. ExecutorService

table Executorsファクトリメソッドの比較

メソッドスレッドプール構成主な用途
newFixedThreadPool(n)固定n本のスレッド負荷を一定に抑えたい場合
newCachedThreadPool()必要に応じて生成・再利用短命な大量タスク
newSingleThreadExecutor()スレッド1本のみ直列実行を保証したい場合
newScheduledThreadPool(n)スケジューリング対応定期実行・遅延実行

description ExecutorServiceの基本的な使い方

スレッドを直接生成・管理する代わりに、ExecutorService にタスクを提出し、スレッドプールで効率よく処理します。必ず shutdown() を呼ぶことが重要です。

// newFixedThreadPool: 固定n本のスレッドプール
ExecutorService fixed = Executors.newFixedThreadPool(2);
fixed.submit(() -> System.out.println("Fixed: " + Thread.currentThread().getName()));
fixed.shutdown();

// newCachedThreadPool: 必要に応じてスレッドを生成・再利用（短命な大量タスク向け）
ExecutorService cached = Executors.newCachedThreadPool();
for (int i = 0; i < 3; i++) {
    final int n = i;
    cached.submit(() -> System.out.println("Cached task-" + n + " by " + Thread.currentThread().getName()));
}
cached.shutdown();

// newSingleThreadExecutor: 1本のスレッドで直列実行を保証
ExecutorService single = Executors.newSingleThreadExecutor();
single.submit(() -> System.out.println("Single: task1"));
single.submit(() -> System.out.println("Single: task2")); // 必ずtask1の後
single.shutdown();

// newScheduledThreadPool: 遅延/定期実行対応
ScheduledExecutorService scheduled = Executors.newScheduledThreadPool(1);
scheduled.schedule(() -> System.out.println("Scheduled after 100ms"), 100, TimeUnit.MILLISECONDS);
scheduled.shutdown();
terminal Output Task A by pool-1-thread-1
Task B by pool-1-thread-2

3. Callable と Future

table Runnable vs Callable の比較

RunnableCallable<V>
抽象メソッドvoid run()V call() throws Exception
戻り値なし任意の型 V
例外検査例外をスローできない検査例外をスローできる
Futureの取得submit()のみ可submit()で取得可

description Futureで非同期処理の結果を取得

Future<V> は非同期タスクの「将来の結果」を表すオブジェクトです。get() を呼ぶと結果が返るまでブロックします。

ExecutorService executor = Executors.newFixedThreadPool(1);

// Callable: 戻り値あり、例外スローが可能
Callable<String> task = () -> {
    Thread.sleep(100);
    return "計算結果: " + (10 + 20);
};

Future<String> future = executor.submit(task);

// future.isDone() : タスクが完了したか確認（ノンブロッキング）
System.out.println("完了済み?: " + future.isDone());

// future.get() : 結果が出るまでブロック
try {
    String result = future.get(); // タイムアウト付き: future.get(1, TimeUnit.SECONDS)
    System.out.println(result);
} catch (InterruptedException | ExecutionException e) {
    e.printStackTrace();
} finally {
    executor.shutdown();
}
terminal Output 完了済み?: false
計算結果: 30

4. 同期制御

table 同期制御の手段比較

手段特徴主な用途
synchronizedメソッドメソッド全体をロック。シンプルインスタンスメソッドの排他制御
synchronizedブロックロック範囲を最小限に絞れるパフォーマンスを重視する場合
LockインタフェースtryLock()など高機能。明示的にlock/unlock複雑な排他制御
volatile可視性のみ保証。複合操作は不可フラグ変数など単純な共有

description synchronized によるスレッドセーフなカウンター

複数スレッドが同じフィールドを同時に読み書きすると「競合状態（Race Condition）」が発生します。synchronized を使うと1スレッドのみがブロック内を実行できます。

public class Counter {
    private int count = 0;

    // synchronizedメソッド: 同時に1スレッドしか実行できない
    public synchronized void increment() {
        count++;
    }

    // synchronizedブロック: ロック粒度を細かくできる
    public void incrementBlock() {
        synchronized (this) {
            count++;
        }
    }

    public synchronized int getCount() { return count; }
}

Counter counter = new Counter();
ExecutorService exec = Executors.newFixedThreadPool(10);
for (int i = 0; i < 1000; i++) {
    exec.submit(counter::increment);
}
exec.shutdown();
exec.awaitTermination(5, TimeUnit.SECONDS);
System.out.println("Count: " + counter.getCount()); // 必ず 1000
terminal Output Count: 1000

5. スレッドセーフなコレクション

table スレッドセーフなコレクションの比較

クラス対応するコレクション特徴
CopyOnWriteArrayListArrayList書き込み時にコピー生成。読み取り多数の場合に有効
ConcurrentHashMapHashMapセグメントロックで高いスループット
ConcurrentLinkedQueueQueueロックフリーで高パフォーマンス
BlockingQueueQueueput/take時にブロック。Producer-Consumerパターン向け
Collections.synchronizedList()任意のList全メソッドをsynchronizedでラップ。旧来の方法

description ConcurrentHashMapの使用例

通常の HashMap はスレッドセーフではなく、並列処理で使うとデータ破損が起こります。ConcurrentHashMap を使うと安全に並列アクセスできます。

// CopyOnWriteArrayList: 書き込み時にコピー。読み取り多数向け
CopyOnWriteArrayList<String> cowList = new CopyOnWriteArrayList<>();
cowList.add("A"); cowList.add("B"); cowList.add("C");
System.out.println("CopyOnWriteArrayList: " + cowList);

// ConcurrentHashMap: セグメントロックで高スループット
ConcurrentMap<String, Integer> map = new ConcurrentHashMap<>();
map.put("apple", 1);
map.put("banana", 2);
map.computeIfAbsent("cherry", k -> k.length()); // キーなし時のみ追加
map.merge("apple", 10, Integer::sum);           // apple: 1+10=11
System.out.println("ConcurrentHashMap: " + map);

// ConcurrentLinkedQueue: ロックフリーで高パフォーマンスなキュー
Queue<String> queue = new ConcurrentLinkedQueue<>();
queue.offer("first"); queue.offer("second");
System.out.println("ConcurrentLinkedQueue poll: " + queue.poll()); // first

// BlockingQueue (LinkedBlockingQueue): put/takeでブロック。Producer-Consumerパターン向け
BlockingQueue<String> bq = new LinkedBlockingQueue<>(10);
bq.put("item1"); // キューが満杯ならブロック
System.out.println("BlockingQueue take: " + bq.take()); // 空ならブロック

// Collections.synchronizedList: 旧来のスレッドセーフラッパー
List<String> syncList = Collections.synchronizedList(new ArrayList<>());
syncList.add("x"); syncList.add("y");
System.out.println("synchronizedList: " + syncList);
terminal Output {apple=11, banana=2, cherry=6}

6. Atomicクラス

table 主なAtomicクラス一覧

クラス対応する型主なメソッド
AtomicIntegerintgetAndIncrement(), compareAndSet()
AtomicLonglongincrementAndGet(), addAndGet()
AtomicBooleanbooleancompareAndSet(expected, update)
AtomicReference<V>任意のオブジェクトcompareAndSet()

description synchronized不要でスレッドセーフなカウンター

AtomicInteger などはCAS（Compare-And-Swap）というCPUの命令を使って、synchronized なしでスレッドセーフな操作を実現します。オーバーヘッドが少なく高速です。

// AtomicInteger: getAndIncrement（現在値を返してから+1）/ compareAndSet（CAS操作）
AtomicInteger ai = new AtomicInteger(0);
System.out.println("getAndIncrement: " + ai.getAndIncrement()); // 0
System.out.println("incrementAndGet: " + ai.incrementAndGet()); // 2
System.out.println("compareAndSet:   " + ai.compareAndSet(2, 100)); // true (2→100)
System.out.println("value:           " + ai.get()); // 100

// AtomicLong: incrementAndGet / addAndGet
AtomicLong al = new AtomicLong(10L);
System.out.println("incrementAndGet: " + al.incrementAndGet()); // 11
System.out.println("addAndGet(5):    " + al.addAndGet(5));      // 16

// AtomicBoolean: compareAndSet
AtomicBoolean flag = new AtomicBoolean(false);
System.out.println("compareAndSet:   " + flag.compareAndSet(false, true)); // true
System.out.println("value:           " + flag.get()); // true

// AtomicReference: 任意のオブジェクトをCAS操作
AtomicReference<String> ref = new AtomicReference<>("initial");
System.out.println("compareAndSet:   " + ref.compareAndSet("initial", "updated"));
System.out.println("value:           " + ref.get()); // updated

// 並列カウントの確認
AtomicInteger atomicCount = new AtomicInteger(0);
ExecutorService exec = Executors.newFixedThreadPool(10);
for (int i = 0; i < 1000; i++) {
    exec.submit(atomicCount::incrementAndGet);
}
exec.shutdown();
exec.awaitTermination(5, TimeUnit.SECONDS);

System.out.println("AtomicInteger count: " + atomicCount.get()); // 必ず 1000

// compareAndSet: 現在値がexpectedと等しい場合のみ更新（CAS操作）
AtomicBoolean flag = new AtomicBoolean(false);
boolean updated = flag.compareAndSet(false, true); // false→trueに変更
System.out.println("Updated: " + updated + ", flag: " + flag.get());
terminal Output AtomicInteger count: 1000
Updated: true, flag: true
