1. 基本定義と4大関数型インタフェース

table 4大基本関数型インタフェースとその派生

インターフェース 抽象メソッド 引数 戻り値 主な用途
Predicate<T> boolean test(T t) 1 boolean 条件判定・フィルタリング
Consumer<T> void accept(T t) 1 なし 出力処理、状態変更
Function<T, R> R apply(T t) 1 任意の型 値の変換、データマッピング
Supplier<T> T get() なし 任意の型 インスタンス生成、遅延評価

BinaryOperator<T> T apply(T t1, T t2) 2 同型 2つの同型データから1つを算出

code 4大基本インタフェースと派生の使用例

各インターフェースをラムダ式で実装した超シンプルな動作例です。

import java.util.function.*;

public class BasicInterfacesDemo {
    public static void main(String[] args) {
        // 1. Predicate (引数1つ → 判定してbooleanを返す)
        Predicate<String> isEmpty = s -> s.isEmpty();
        System.out.println("Predicate: " + isEmpty.test("")); // true

        // 2. Consumer (引数1つ → 処理するが戻り値はなし)
        Consumer<String> printer = s -> System.out.println("Consumer: " + s);
        printer.accept("Hello"); // Consumer: Hello

        // 3. Function (引数1つ → 別の型に変換して返す)
        Function<String, Integer> length = s -> s.length();
        System.out.println("Function: " + length.apply("Java")); // 4

        // 4. Supplier (引数なし → 生成・評価して任意の値を返す)
        Supplier<Double> random = () -> Math.random();
        System.out.println("Supplier: " + random.get()); // 例: 0.846...

        // 5. UnaryOperator (引数1つ、戻り値がすべて同じ型)
        UnaryOperator<String> upper = s -> s.toUpperCase();
        System.out.println("UnaryOperator: " + upper.apply("gold")); // GOLD

        // 6. BinaryOperator (引数2つ、戻り値がすべて同じ型)
        BinaryOperator<Integer> add = (a, b) -> a + b;
        System.out.println("BinaryOperator: " + add.apply(10, 20)); // 30

        // 7. BiFunction (引数2つ → 別の型に変換して返す)
        BiFunction<String, Integer, String> repeat = (s, n) -> s.repeat(n);
        System.out.println("BiFunction: " + repeat.apply("A", 3)); // AAA
    }
}

description 関数型インタフェースの定義ルール

関数型インタフェース（SAM: Single Abstract Method）とは、抽象メソッドをただ1つだけ持つインターフェースのことです。
default メソッドや static メソッド、また Object クラスに定義されているパブリックメソッド（toString() や equals() など）のオーバーライド宣言は、抽象メソッドの数にカウントされません。
アノテーション @FunctionalInterface を付与すると、ルールを満たしていない場合にコンパイルエラーになるため、付与することが推奨されます（試験ではアノテーションがなくてもSAMを満たしていればラムダ式に代入可能です）。 

@FunctionalInterface
interface CustomChecker {
    boolean check(String value); // 抽象メソッド1つのみ
    
    default void printHello() { System.out.println("Hello"); } // defaultはカウント外
    static void printStatic() { System.out.println("Static"); } // staticもカウント外
}

public class Main {
    public static void main(String[] args) {
        // ラムダ式での実装
        CustomChecker checker = (s) -> s != null && s.length() > 5;
        System.out.println("Checker Result: " + checker.check("JavaGold"));
    }
}
terminal Output Checker Result: true

description 記述ルールと型推論の一貫性

ラムダ式を書く際、引数の省略や型推論には一貫性が求められます。
例えば、引数が2つ以上ある場合に、片方だけに型を書いて片方を省略する（例: (String a, b) -> ...）ことはできません。すべて型を書くか、すべて省略するかのどちらかでなければなりません。
また、アノテーションをラムダ式の引数に付けたい場合は、必ず var または明示的な型指定が必要になります（例: (@NotNull var s) -> ...）。 

// 正しいラムダ式パターン
BiFunction<Integer, Integer, Integer> calc1 = (var x, var y) -> x + y; // varの正しい使用
Function<String, Integer> len1 = (String s) -> { return s.length(); }; // 完全な記述
Function<String, Integer> len2 = s -> s.length(); // 最大限の省略記述

System.out.println("calc1: " + calc1.apply(10, 20));
System.out.println("len2: " + len2.apply("Lambda"));
terminal Output calc1: 30
len2: 6

3. メソッド参照 (Method References)

table メソッド参照の4パターン比較

パターン 構文 ラムダ式での等価表現 具体例
静的メソッドの参照 ClassName::staticMethod (args) -> ClassName.staticMethod(args) Integer::parseInt
特定オブジェクトの
インスタンスメソッド参照		 instance::instanceMethod (args) -> instance.instanceMethod(args) System.out::println
任意オブジェクトの
インスタンスメソッド参照		 ClassName::instanceMethod (obj, args) -> obj.instanceMethod(args) String::isEmpty
コンストラクタ参照 ClassName::new (args) -> new ClassName(args) ArrayList::new

description 任意オブジェクトと特定オブジェクトの違い（最頻出）

メソッド参照 :: を使うと、ラムダ式をさらに簡潔に記述できます。
試験で最も引っかけられやすいのが、「特定オブジェクト」と「任意オブジェクト」のインスタンスメソッドの参照の違いです。
例えば、String::isEmpty は「任意のStringインスタンスに対してメソッドを実行する」という意味になるため、ラムダ式で表すと第1引数がレシーバ（呼び出し対象インスタンス）になり、(s) -> s.isEmpty() と等価になります。そのため、関数型インタフェースには「レシーバとなる引数」を1つ受け取る定義が必要です。 

// 1. 静的メソッド参照 (Function は引数を受け取り結果を返す)
Function<String, Integer> parser = Integer::parseInt; 
System.out.println("Static parse: " + (parser.apply("123") + 10));

// 2. 任意オブジェクトのインスタンスメソッド参照
// (引数の第1引数がメソッドを呼び出すインスタンス（String）になる)
BiPredicate<String, String> checker = String::startsWith; 
System.out.println("StartsWith: " + checker.test("JavaGold", "Java"));

// 3. コンストラクタ参照
Supplier<StringBuilder> sbCreator = StringBuilder::new;
StringBuilder sb = sbCreator.get().append("Constructor Ref");
System.out.println(sb.toString());
terminal Output Static parse: 133
StartsWith: true
Constructor Ref

4. 変数キャプチャと「効果的にfinal」

description 「効果的にfinal (effectively final)」とは

ラムダ式内で外側のスコープの「ローカル変数」を参照することを **変数キャプチャ** と呼びます。
このキャプチャ対象となるローカル変数は、明示的に final と宣言されているか、あるいは**一度も値が変更されていない（効果的にfinal）**状態でなければなりません。
メソッドの中でローカル変数に対して一度でも再代入が行われると、その変数は効果的にfinalとは見なされず、ラムダ式の中で参照した時点でコンパイルエラーが発生します。 

public class CaptureDemo {
    private int instanceVal = 100; // インスタンスフィールド

    public void run() {
        int localVal = 50; // finalがついていないが、一度も変更されていない（効果的にfinal）
        int nonFinalVal = 10;
        nonFinalVal = 20; // ここで再代入したため、効果的にfinalではなくなった！

        Consumer<String> consumer = (msg) -> {
            System.out.println(msg + " Local: " + localVal); // ⭕️ 参照可能 (effectively final)
            
            // System.out.println(nonFinalVal); // ❌ コンパイルエラー！(effectively finalではない)
            
            instanceVal += 50; // ⭕️ インスタンス変数は変更可能！
            System.out.println("Instance modified: " + instanceVal);
        };
        consumer.accept("Test");
    }
}

// 実行確認
new CaptureDemo().run();
terminal Output Test Local: 50
Instance modified: 150

5. プリミティブ型に特化した関数型インタフェース

table プリミティブ特化型インタフェースの一覧

ボクシング/アンボクシングのオーバーヘッド（無駄なオブジェクト生成）を避けるため、各プリミティブ型（int, long, double）に特化したインターフェースが提供されています。

基本型 型特化 Predicate 型特化 Consumer 型特化 Supplier Function系（引数がプリミティブ） Function系（戻り値がプリミティブ）
int IntPredicate IntConsumer IntSupplier IntFunction<R> ToIntFunction<T>
long LongPredicate LongConsumer LongSupplier LongFunction<R> ToLongFunction<T>
double DoublePredicate DoubleConsumer DoubleSupplier DoubleFunction<R> ToDoubleFunction<T>

description プリミティブ特化型のメリットと命名規則

Predicate<Integer> を使って数値判定を行うと、要素を検証するたびに「基本データ型の int をラッパークラスの Integer に包む」という暗黙のボクシングが発生します。
これに対し、IntPredicate を使用すると内部的には int のまま処理が流れるため、メモリや計算性能が最適化されます。
試験対策のポイント：「引数がプリミティブで戻り値がオブジェクト」のパターン（例: IntFunction<R>）と、「引数がオブジェクトで戻り値がプリミティブ」のパターン（例: ToIntFunction<T>）の命名規則、およびそれぞれのメソッド名（applyAsInt などの As〇〇 がつくメソッド）を区別できるようにしてください。 

// 1. 引数が int で、戻り値がString型
IntFunction<String> intFunc = (i) -> "Value is " + i;
System.out.println("IntFunction: " + intFunc.apply(10)); // 通常通り apply

// 2. 引数がStringで、戻り値が int型
ToIntFunction<String> toIntFunc = (s) -> s.length();
System.out.println("ToIntFunction: " + toIntFunc.applyAsInt("JavaGold")); // メソッド名が applyAsInt になる！

// 3. 2つの引数をプリミティブ double で受け取り計算
DoubleBinaryOperator addDouble = (d1, d2) -> d1 + d2;
System.out.println("DoubleBinaryOperator: " + addDouble.applyAsDouble(1.5, 2.5)); // applyAsDouble
terminal Output IntFunction: Value is 10
ToIntFunction: 8
DoubleBinaryOperator: 4.0
